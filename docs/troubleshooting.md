# Troubleshooting

## "content is required and must be a string" (400 error)

**Symptom**: API returns `400` with `{"error": "content is required and must be a string"}` when creating a page.

**Cause**: The POST body is missing the `content` field, or `content` is not a string.

**Fix**: Ensure the `Content-Type: application/json` header is set, and the request body includes a `content` field with a string value:

```bash
curl -X POST http://localhost:3000/api/pages \
  -H "Content-Type: application/json" \
  -d '{"content": "## Review\n\n{{approve:ok|label=Approve?}}"}'
```

Common mistakes:
- Sending form-encoded data instead of JSON (missing `-H "Content-Type: application/json"`)
- Sending `{"content": null}` or `{"content": 123}` instead of a string

## "page not found" (404 on API or User server)

**Symptom**: API returns `{"error": "page not found"}` or User server returns `"Page not found"`.

**Cause**: The page ID is invalid, or the page was never created.

**Fix**: Verify the page ID. Page IDs are case-sensitive 10-character strings generated by nanoid. Double-check for typos, trailing whitespace, or URL-encoding issues.

## URLs in API response point to localhost in production

**Symptom**: `POST /api/pages` returns URLs like `http://localhost:3001/p/aBcDeFgHiJ` even though the server is deployed.

**Cause**: The `USER_BASE_URL` environment variable is not set.

**Fix**: Set `USER_BASE_URL` to the public URL of the user-facing server:

```bash
export USER_BASE_URL=https://hitl.example.com
```

With Docker:

```bash
docker run -e USER_BASE_URL=https://hitl.example.com ...
```

## Form shows "This page has already been submitted"

**Symptom**: Human opens a page URL but sees "This page has already been submitted" instead of the form.

**Cause**: The page was already responded to. Each page can only be submitted once.

**Fix**: This is expected behavior by design. Pages are single-use. If you need a new response, have the agent create a new page with `POST /api/pages`.

## Approve/Reject buttons don't work

**Symptom**: Clicking Approve or Reject does nothing — the button doesn't highlight, and the form submits with an empty value for the approve field.

**Cause**: JavaScript is disabled or blocked in the browser.

**Fix**: Ensure JavaScript is enabled. The approve/reject buttons require JavaScript to set the hidden form field value. All other field types (text, select, checkbox) work without JavaScript.

## Database errors on startup

**Symptom**: Application crashes on startup with an error related to SQLite or the database file.

**Cause**: The directory specified in `DB_PATH` doesn't exist, or the process doesn't have write permission.

**Fix**: Ensure the parent directory of `DB_PATH` exists and is writable:

```bash
# Default path
mkdir -p ./data

# Custom path
mkdir -p /var/lib/hitl
```

For Docker, mount a volume to `/app/data`:

```bash
docker run -v hitl-data:/app/data ...
```

## `npm install` fails with native module errors

**Symptom**: `npm install` fails with compilation errors mentioning `better-sqlite3`, `node-gyp`, or `g++`.

**Cause**: `better-sqlite3` is a native module that requires a C++ compiler to build.

**Fix**: Install build tools for your platform:

```bash
# Debian / Ubuntu
sudo apt-get install build-essential

# macOS
xcode-select --install

# Alpine Linux
apk add build-base python3
```

Then re-run `npm install`.

---

## Port already in use

**Symptom**: Application fails to start with `EADDRINUSE` error on port 3000 or 3001.

**Cause**: Another process is already using the port.

**Fix**: Either stop the conflicting process, or set different ports via environment variables:

```bash
export API_PORT=8080
export USER_PORT=8081
npm start
```

To find what's using the port:

```bash
lsof -i :3000
```
